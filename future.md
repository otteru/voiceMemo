# 향후 개선 계획

## STT 처리 방식 비교

### 방식 1: 파일 업로드 후 스트리밍 STT (현재 방식)

```
브라우저 녹음 → Blob 생성 → HTTP POST → 서버 저장 → transcribe_file()
                                                    ↓
                                        파일을 청크로 나눠서 스트리밍 STT
```

**장점:**
- ✅ 프론트엔드 구현이 간단함
- ✅ 녹음 파일 백업 가능
- ✅ 실패 시 재처리 가능

**단점:**
- ❌ 녹음 완료 후에만 처리 시작
- ❌ 파일 전송 시간 소요
- ❌ 스토리지 사용

**필요 패키지:**
- `python-multipart` (HTTP 파일 업로드)

---

### 방식 2: 실시간 WebSocket 스트리밍 (미래 개선안)

```
브라우저 녹음 → 실시간 청크 → WebSocket → stream_transcribe()
                                             ↓
                                    실시간 STT 결과 반환
```

**장점:**
- ✅ 진짜 실시간 처리
- ✅ 녹음 중 텍스트 확인 가능
- ✅ 파일 저장 불필요 (스토리지 절약)

**단점:**
- ❌ 프론트엔드/백엔드 구현 복잡
- ❌ WebSocket 연결 관리 필요
- ❌ 네트워크 불안정 시 재처리 어려움

**필요 패키지:**
- `websockets` (이미 설치됨)

---

## 결론

- **현재**: 방식 1 구현 (안정적, 간단)
- **향후**: 방식 2로 전환 검토 (실시간성 개선)
